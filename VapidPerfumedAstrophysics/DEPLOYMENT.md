# Ria — Deployment Guide

This codebase is split into three independently deployable pieces:

```
/frontend   → Vercel (or Netlify)   — React PWA static build
/backend    → Render                — Express REST API
Database    → Supabase              — PostgreSQL + Storage
```

---

## 1. Database — Supabase

**Free tier:** 500 MB storage, 2 GB bandwidth/month, project pauses after 7 days of inactivity on free tier.

### a. PostgreSQL setup

1. Create a project at [supabase.com](https://supabase.com)
2. Go to **Settings → Database** and copy the **Connection string** (URI format)
3. Run the schema migration:
   ```sh
   cd backend
   DATABASE_URL="your-supabase-connection-string" node src/db/migrate.js
   ```

### b. Bootstrap first admin (seed script)

Do **not** manually insert admins via raw SQL — the password hash must be created by the app's bcrypt logic. Use the idempotent seed script instead:

```sh
# From repo root — safe to re-run; does nothing if an admin already exists
DATABASE_URL="your-supabase-url" BOOTSTRAP_ADMIN_PASSWORD="YourStrongPassword!" node scripts/seed-admin.js
```

This creates the founding global admin (Emmanuel Chesco / Rotaract Muhimbili) with the password you supply. All values can be overridden via environment variables — see `scripts/seed-admin.js` for the full list.

### c. Supabase Storage (profile pictures & broadcast images)

Profile pictures and broadcast images use Supabase Storage's free-tier bucket.

**Free tier limit:** 1 GB storage, 2 GB bandwidth/month.

Setup:
1. In your Supabase project, go to **Storage**
2. Create a new bucket named `ria-media`
3. Set the bucket to **public** (images are served by URL, no auth needed to read)
4. Copy your project URL and anon key from **Settings → API**
5. Set these env vars on Render:

```
SUPABASE_URL=https://your-project.supabase.co
SUPABASE_ANON_KEY=your-anon-key
```

**Cost awareness:** Both profile pictures and broadcast images are compressed client-side before upload (profile pics: max 512px, WebP 0.78; broadcast images: max 1200px, WebP 0.82). This keeps the free-tier storage bucket lasting as long as possible. Monitor usage in the Supabase dashboard. The System Health widget in the admin dashboard surfaces DB size; add a Supabase Storage usage alert separately when you hit ~800 MB.

**Free-tier limits to watch:**
- DB storage: 500 MB (monitored by the System Health widget)
- Storage bucket: 1 GB (not currently in the health widget — add if it becomes a concern)
- Bandwidth: 2 GB/month
- Connections: 60 simultaneous
- Project pauses after 7 days of inactivity (upgrade to pro to disable)

## 2. Backend — Render

**Free tier:** 750 hours/month, spins down after 15 minutes of inactivity (cold start ~30s).

1. Push code to GitHub (the whole repo, or just `/backend`)
2. Create a new **Web Service** on [render.com](https://render.com)
3. Set **Root Directory** to `backend`
4. Set **Build Command**: `npm install`
5. Set **Start Command**: `node src/server.js`
6. Set environment variables in Render dashboard:

| Variable | Value |
|---|---|
| `DATABASE_URL` | Your Supabase connection string |
| `JWT_SECRET` | Random 64-char string |
| `JWT_REFRESH_SECRET` | Different random 64-char string |
| `FRONTEND_URL` | Your Vercel frontend URL (e.g. `https://ria.vercel.app`) |
| `NODE_ENV` | `production` |
| `PORT` | (auto-set by Render, do not override) |
| `SUPABASE_URL` | Your Supabase project URL |
| `SUPABASE_ANON_KEY` | Your Supabase anon key |

**Note:** On the free tier, the backend spins down after 15 minutes of inactivity. The first request after a cold start will take ~30 seconds. Field users with offline data will not notice (data queues locally and syncs when the server wakes up).

**Free-tier limits to watch:**
- 750 hours/month shared across all your Render services (sufficient for ~1 service)
- 512 MB RAM
- Shared CPU

## 3. Frontend — Vercel

**Free tier:** Unlimited bandwidth on hobby projects, no build time limits for personal use.

1. Push the `/frontend` directory to a GitHub repo (or the whole monorepo)
2. Import the project at [vercel.com](https://vercel.com)
3. Set **Root Directory** to `frontend`
4. Set **Framework Preset** to `Vite`
5. Set environment variables in Vercel dashboard:

| Variable | Value |
|---|---|
| `VITE_API_URL` | Your Render backend URL (e.g. `https://ria-api.onrender.com/api`) |

6. Deploy. Vercel will auto-deploy on every push to `main`.

**PWA notes:**
- The service worker and manifest are generated by `vite-plugin-pwa` at build time
- iOS "Add to Home Screen" works via Safari's share sheet — no additional setup needed
- Android Chrome will show the native install prompt automatically

**Free-tier limits to watch:**
- 100 GB bandwidth/month (generous)
- No paid plan needed for this scale
- Vercel may inject a "Powered by Vercel" badge in the page footer on hobby plans — check current terms

## 4. Environment Variables Reference

### Backend (`/backend/.env`)
```env
DATABASE_URL=postgresql://user:password@host:5432/postgres
JWT_SECRET=<64-char random string>
JWT_REFRESH_SECRET=<different 64-char random string>
FRONTEND_URL=https://your-frontend.vercel.app
PORT=3001
NODE_ENV=development
BOOTSTRAP_ADMIN_PASSWORD=<set at deploy time, never commit>
SUPABASE_URL=https://your-project.supabase.co
SUPABASE_ANON_KEY=your-supabase-anon-key
```

### Frontend (`/frontend/.env.local`)
```env
VITE_API_URL=http://localhost:3001/api
```

For production, set `VITE_API_URL` to your Render backend URL in the Vercel environment settings.

## 5. Database Backup

To protect project data from free-tier pauses or service disappearance, set up a scheduled database backup:

**Option A — Supabase built-in (Pro tier):** Daily backups with point-in-time recovery.

**Option B — Manual/scheduled script (free):**
```sh
# Add to a cron job or GitHub Actions scheduled workflow
pg_dump $DATABASE_URL | gzip > backup_$(date +%Y%m%d).sql.gz
# Upload to S3, Google Drive, or any object storage
```

A GitHub Actions workflow running nightly `pg_dump` to an S3 bucket or Google Drive is sufficient for this scale and cost.

## 6. Third-party Services & Free-tier Limits Summary

| Service | What it does | Free limit | When to upgrade |
|---|---|---|---|
| Supabase | PostgreSQL database | 500 MB storage, pauses after 7d inactivity | If DB > 400 MB or you need always-on |
| Supabase Storage | Profile pics & broadcast images | 1 GB storage, 2 GB bandwidth | If media storage > ~800 MB |
| Render | Express API | 750 hrs/month, spins down on idle | If cold start latency is unacceptable |
| Vercel | React PWA hosting | 100 GB bandwidth/month | Almost never at this scale |

## 7. Post-deploy Checklist

- [ ] Schema migration run: `node src/db/migrate.js`
- [ ] Bootstrap admin created: `node scripts/seed-admin.js`
- [ ] Backend health check passing: `curl https://your-api.onrender.com/api/health`
- [ ] Frontend loads and shows splash screen
- [ ] Register a field user and log in
- [ ] Create a project and questionnaire in admin dashboard
- [ ] Submit a test questionnaire on mobile (online + offline)
- [ ] Verify sync works after going offline then online
- [ ] Test "Add to Home Screen" on iOS Safari and Android Chrome
- [ ] Confirm System Health widget shows DB stats in admin
- [ ] Set up database backup schedule
- [ ] Supabase Storage bucket "ria-media" created and set to public
- [ ] Test a broadcast from admin dashboard (Broadcasts section)
- [ ] Verify leaderboard populates after a few test submissions
